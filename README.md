# Week 7 Homework - Terraform on GCP

  

## What this does

  

I worked with my group Hungry Wolves 🐺 to set up a VPC network on Google Cloud Platform using Terraform. I also creates a local text file using the local provider just to demo how that works. Outputs the VPC name at the end.

  

## How I did it

  

Started by setting up a GCP project (`seir-1-490220`) and creating a service account with the right IAM permissions (Compute Admin, basically). Downloaded the JSON key for that service account and set the `GOOGLE_APPLICATION_CREDENTIALS` env variable to point at it so Terraform could authenticate.

  

Then wrote out the `.tf` files one by one:

  

- `0-providers.tf` - declared the `hashicorp/google` and `hashicorp/local` providers. Pinned google to `~> 7.0` since that's the latest major version on the registry.

- `1-vpc.tf` - VPC network and subnet. Based this off the example in the Terraform registry for `google_compute_network`. Had to set `auto_create_subnetworks = false` because I want to control my own subnets. The subnet uses a `/16` CIDR with a secondary range.

- `2-lodcalfile.tf` - creates a `foo.bar` file with my favorite food using `local_file`. Simple resource, just testing that the local provider works.

- `output.tf` - outputs the VPC name and the file content so you can see them after `terraform apply`.

  

Ran `terraform init` first to pull down the providers, then `terraform plan` to check everything looked right, then `terraform apply`.

## Issues I ran into

  

- The registry example for `google_compute_network` includes two separate network resources. When I copied the example I ended up with both a `week7_vpc` and a leftover `custom-test` network in my code. The subnet was referencing `custom-test` instead of `week7_vpc`. Fixed by removing the extra network and updating the subnet to reference the right one.

  

- Initially forgot to set `auto_create_subnetworks = false` on the VPC and terraform plan warned me that GCP would create default subnets in every region, which I didn't want.

  

- Had to make sure the service account had the right permissions - first apply failed because Compute API wasn't enabled on the project yet. Enabled it through the GCP console and re-ran.

  

- The output block was returning the entire VPC resource object instead of just the name. Fixed by changing `google_compute_network.week7_vpc` to `google_compute_network.week7_vpc.name` in output.tf.

  

- Started with all the terraform files at the repo root instead of inside the `infra/` folder. Moved everything in and re-ran `terraform init` from inside `infra/`. The state carried over fine since the GCP resource IDs didn't change.

  

## Security issues I ran into

  

- The `.gitignore` originally had `.json` instead of `*.json`. The `.json` pattern only matches a file literally named `.json`, so the service account key (`seir-1-490220-31b0e1037032.json`) was not actually being ignored and would have been committed to GitHub with the credentials in it. Fixed by correcting the pattern to `*.json`.

  

- After moving the terraform files into `infra/`, the `.gitignore` moved with them. That left the JSON key at the repo root with no gitignore coverage. Fixed by adding a second `.gitignore` at the repo root that also covers `*.json`.

  

- Added `lifecycle { prevent_destroy = true }` to the `local_file` resource so that running `terraform destroy` doesn't delete `foo.bar`. To do a full teardown you have to remove it from state first with `terraform state rm local_file.favorite_food` then run destroy.

## Running it

  

```bash

export GOOGLE_APPLICATION_CREDENTIALS="/path/to/your-key.json"

cd infra

terraform init

terraform plan

terraform apply

```

  

To destroy GCP resources while keeping foo.bar:

```bash

terraform state rm local_file.favorite_food

terraform destroy

```
